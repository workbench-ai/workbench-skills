# Workflow Authoring

This is the canonical guide for authoring Workbench workflow YAML. It is written for both humans and LLMs. If you need to know which fields are valid, how runtime interprets them, or how to validate a file before saving it, start here.

Workflow definitions are YAML-first. The source of truth is the document stored in `.workbench/workflows/*.yaml`, whether it was created in an editor, generated by an LLM, or saved through the web UI. The web authoring surface edits that YAML directly; it does not create a second workflow format.

When you are working from a git repo, prefer the CLI helpers around this guide:

- `workbench init .` creates the minimal repo-local `.workbench` scaffold.
- `workbench template list` shows the bundled starter workflows.
- `workbench init . --template <starter>` installs one bundled example into `.workbench/workflows/`.
- `workbench validate` validates the repo's workflow collection and, by default, checks whether the current local platform is ready to run each structurally valid workflow.
- `workbench validate --schema-only` runs the same shared contracts validator used by the runtime save path and web editor.

## Start Here

Workflow files must be a single YAML document under `.workbench/workflows/*.yaml`. The required top-level sections are:

- `state`
- `triggers`
- `workspace`
- `hooks`
- `stages`

The optional top-level sections are:

- `metadata`
- `inputs`
- `actions`

Important rules:

- `stages` is required, but it may be an empty array. Use `stages: []` for explicit hook-only workflows.
- `harness` is required when `stages` contains at least one stage. There is no default harness.
- `harness.auth` is required when a `harness` block is present and must be an object. The installed harness package validates the object's contents. `harness.config` defaults to `{}`.
- Manual-only workflows still need `triggers: {}`.
- Top-level `inputs` are reusable definitions. They do nothing until a manual trigger or action binds them by `ref`.
- Unknown core keys are invalid unless they are under `metadata` or prefixed with `x-`.
- `x-*` extension keys are allowed anywhere, ignored by Workbench core, and preserved when the API rewrites canonical YAML.
- Duplicate effective workflow ids are invalid across the whole workflow collection, even when filenames differ.
- Zero-stage workflows are hook-only. They do not start a harness session and they do not support same-execution continuation turns.
- If you want native terminal chat, do not author a workflow for it. Use the runtime's first-class chat catalog exposed from `GET /api/v1/chat/catalog` and start threads with `POST /api/v1/chat/executions` using `harness_id` plus `workspace_mode`.

Validated examples:

- [Codex review loop](examples/workflows/codex-review-loop.yaml)
- [Claude profile triage](examples/workflows/claude-profile-triage.yaml)
- [Hook-only deterministic](examples/workflows/hook-only-deterministic.yaml)
- [Interactive conversation](examples/workflows/interactive-conversation.yaml)
- [Manual review loop](examples/workflows/manual-review-loop.yaml)
- [Webhook Claude gated](examples/workflows/webhook-claude-gated.yaml)

The dedicated web `/chat` route is backed by the first-class runtime chat catalog rather than by authored workflow YAML. If you want workflow-owned back-and-forth with explicit resume semantics, use the interactive conversation pattern above: start the first turn through `POST /api/v1/workflows/:id/executions`, then append later turns to the same execution with `POST /api/v1/executions/:id/triggers/manual` while the stage uses `session: resume`.

## Minimal Valid Workflow

```yaml
metadata:
  name: Simple Task

state:
  initial: in_progress
  terminal:
    - done
    - failed
    - canceled
  on_attempt:
    completed: done
    failed: failed
    canceled: canceled

triggers: {}

workspace: {}

hooks:
  after_create: |
    git init .
    git config user.email workbench@example.com
    git config user.name "Workbench"
    printf 'hello from workbench\n' > result.txt
    git add result.txt
    git commit -m "seed execution workspace" || true

stages:
  - id: implement
    prompt: |
      Create result.txt with the text "hello from workbench".

harness:
  id: openai/codex
  auth:
    strategy: secret_ref
    ref: OPENAI_API_KEY
  model: gpt-5.4
  effort: medium
  config: {}
```

This is valid because `workspace` and `hooks` may be partial objects. `workspace.mode` defaults to `managed`, `workspace.prune_ttl_seconds` defaults to `604800`, and `hooks.timeout_ms` plus harness `retry` and `cancel` have defaults. Harness-specific auth/config validation is delegated to the installed harness manifest, and the `after_create` hook makes the empty managed execution workspace git-backed before the first stage runs. `triggers: {}` means the workflow is manual-only; manual execution does not require a cron or webhook trigger.

## Minimal Hook-Only Workflow

```yaml
metadata:
  name: Deterministic Hooks Only

state:
  initial: queued
  terminal:
    - done
    - failed
    - canceled
  on_attempt:
    completed: done
    failed: failed
    canceled: canceled

triggers: {}

workspace: {}

hooks:
  after_create: |
    git init .
    git config user.email workbench@example.com
    git config user.name "Workbench"
    printf 'seed\n' > README.md
    git add README.md
    git commit -m "seed workspace"
  before_run: |
    mkdir -p artifacts
    printf 'deterministic result\n' > artifacts/result.txt

stages: []
```

This form is valid because `stages` is still present, but empty. With no stages, Workbench runs `after_create`, `before_run`, and `after_run` without starting a harness session. If hooks create a git repo with a valid `HEAD` before completion, change summaries and previews use that baseline. If they do not, the workflow still completes, but diff-backed inspection stays empty.

## Effective Workflow ID And File Naming

Every loaded workflow has one effective workflow id:

- If `metadata.id` is present, that is the effective id.
- Otherwise the filename stem is the effective id.

Examples:

- `.workbench/workflows/review.yaml` with no `metadata.id` loads as `review`.
- `.workbench/workflows/review.yaml` with `metadata.id: security-review` loads as `security-review`.

Additional create-time behavior matters when authoring through the API or web UI:

- Creating a new workflow without `metadata.id` derives the filename from `metadata.name` by slugifying it. If `metadata.name` is absent or slugifies to an empty value, the fallback base is `workflow`.
- Updating an existing workflow without `metadata.id` keeps the existing file path and therefore keeps the current filename-derived id.

`file_path` and effective workflow id can therefore drift if someone edits `metadata.id` inside an existing file without renaming the file. Validation still cares about the effective id, not the filename.

## Top-Level Reference

### `metadata`

`metadata` is optional.

- `id`: optional string override for the effective workflow id
- `name`: optional display name
- `enabled`: optional boolean, defaults to `true`
- `description`: optional string or `null`
- any additional keys are allowed and preserved

`metadata.enabled: false` hides the workflow from manual-run surfaces and prevents new trigger-driven executions, but the file still loads as a valid workflow.

### `state`

`state` is required and defines execution-level business state, not runtime lifecycle state.

- `initial`: required string
- `terminal`: required non-empty array of strings
- `on_attempt`: optional object with `completed`, `failed`, and `canceled`

Runtime uses these values in several places:

- `execution.workflow_state` starts at `state.initial`.
- When an attempt finishes with no further retry, Workbench may move `execution.workflow_state` to `state.on_attempt.completed`, `state.on_attempt.failed`, or `state.on_attempt.canceled`.
- Runtime separately manages `execution.lifecycle_state` such as `queued`, `running`, `waiting`, or terminal outcomes. The board uses workflow state; badges and activity surfaces use lifecycle state.
- The board state vocabulary is derived in this order: `state.initial`, non-terminal `state.on_attempt.*`, non-terminal `actions[].set_state`, then terminal states in declared order.

### `inputs`

`inputs` is optional and defines reusable user-input descriptors for manual runs and actions.

Shared fields:

- `id`: required string, unique within the workflow
- `label`: required string
- `description`: optional string or `null`
- `required`: optional boolean, defaults to `false`
- `type`: required `text`, `textarea`, or `select`

`text` and `textarea` inputs need no additional fields.

`select` inputs require `options`, which support exactly one source:

- static `values`: a non-empty array of strings or `{ value, label?, description? }`
- dynamic `command`: a shell command plus optional `timeout_ms`, default `10000`

Static select options may also set `default_value`, which must match one of the declared option values.

Command-backed selects must print JSON to stdout in this shape:

```json
{
  "options": [
    { "value": "main", "label": "main" },
    { "value": "release/2026.03", "label": "release/2026.03" }
  ],
  "default_value": "main"
}
```

Rules:

- Duplicate `inputs[].id` values are invalid.
- Duplicate select-option values are invalid.
- A command-backed select that returns duplicate values or an unknown `default_value` is rejected at resolution time.
- Input definitions are reusable. Bind them into `triggers.manual.inputs` or `actions[].inputs` instead of redefining the same field twice.

### `triggers`

`triggers` is required and is always an object. It may be empty.

- `manual`: optional object `{ inputs }`
- `cron`: optional array of `{ schedule, timezone, payload? }`
- `webhook`: optional object `{ hmac_secret_env }`

Notes:

- Manual execution is always available for loaded, enabled workflows through `POST /api/v1/workflows/:id/executions`, which returns `201` with the persisted queued `ExecutionDetail`.
- `triggers.manual` does not enable or disable manual execution. It only declares generated input bindings for the UI and the resolver endpoints.
- `triggers.manual.inputs` is an array of `{ ref, payload_key }` bindings. `ref` must match a top-level input id, and `payload_key` is where the resolved value is written inside `TriggerEvent.payload`.
- First-party same-execution continuation is available through `POST /api/v1/executions/:id/triggers/manual`, which accepts the same `{ payload }` shape, appends another trigger onto the existing execution without creating a new attempt, and returns `200` with the updated queued `ExecutionDetail`.
- Native chat threads start via `POST /api/v1/chat/executions`, not via workflow execution creation. Once the session is live, follow-up input goes through the execution-scoped session APIs instead of `POST /api/v1/executions/:id/triggers/manual`.
- `cron[].payload`, verified webhook JSON bodies minus reserved routing fields such as top-level `execution_id`, and manual-run payloads all become `TriggerEvent.payload`.
- When `triggers.manual.inputs` is present, the web UI resolves those input descriptors first and then compiles the submitted values back into the normal `{ payload }` request body. Workflows without manual input bindings still accept arbitrary JSON objects, but the web UI now keeps the default empty object implicit and exposes the raw JSON editor only under `Advanced options`.
- A verified webhook with `execution_id` continues the same non-terminal execution as a new turn. It appends trigger history and resumes the current attempt in place; it does not create a new attempt.
- Webhook payloads must be JSON objects. Arrays, primitives, and empty JSON bodies are rejected.

### `workspace`

`workspace` is required.

- `mode`: optional `managed` or `project`, defaults to `managed`
- `prune_ttl_seconds`: optional positive integer, defaults to `604800`

Managed execution workspaces are created by Workbench at:

`~/.workbench/workspaces/<project-bucket>/<ownerId>/<executionId>/workspace`

Each execution gets exactly one mutable workspace. In `managed` mode, that workspace starts empty, lives outside the project root, and is reused across continuation turns, retries, and action-triggered reruns. In `project` mode, the execution workspace is the repo root at `WB_PROJECT_ROOT`, so commands run directly in the active checkout. Workbench derives `<project-bucket>` from the source project checkout path so different projects do not collide in managed mode, and `<ownerId>` is the workflow id for workflow-owned executions. `workspace.root` is no longer valid workflow YAML; legacy files that still declare it fail validation with an explicit migration error.

### `hooks`

`hooks` is required.

- `after_create`: string or `null`
- `before_run`: string or `null`
- `after_run`: string or `null`
- `before_remove`: string or `null`
- `timeout_ms`: optional positive integer, defaults to `60000`
- `env`: optional string-to-string map

Hook commands run in the execution workspace. `after_create` runs only when the execution workspace directory is first created. `before_run` and `after_run` run for each attempt, including attempt 1 and later retry or action-created attempts. Pure continuation turns do not create a new attempt, so they do not rerun attempt lifecycle hooks. `before_run` failure aborts the attempt. `after_run` and `before_remove` failures are recorded as warnings. `before_remove` runs once before the execution workspace is pruned.

In `project` mode, Workbench does not create or prune the workspace path, so `after_create` and `before_remove` do not run. If a workflow needs the current project files while staying in `managed` mode, `after_create` can seed the empty execution workspace by running `git clone "$WB_PROJECT_ROOT" .`. Scratch workflows can instead use `git init .` and create their own starting files. If the workflow should work directly in the active checkout, set `workspace.mode: project` and keep `after_create` `null`. Retry-safe setup that depends on the source repository state, such as `git worktree add`, belongs in `before_run` rather than `after_create`.

Managed stageful workflows must leave the workspace git-backed with a valid `HEAD` before the first stage runs. Zero-stage workflows relax only that last requirement: if hooks never create a git repo or commit, the attempt still completes, but `execution.start_git_ref` stays `null`, so diff-backed change summaries and previews have no baseline.

### `stages`

`stages` is required and may be empty. Array order is execution order when stages exist.

`stages: []` declares a hook-only workflow. In that mode:

- Workbench still runs `after_create`, `before_run`, and `after_run`.
- `harness` becomes optional.
- Same-execution manual or webhook continuation is not supported.

For non-empty stage lists, each stage supports:

- `id`: required string, unique within the workflow
- `prompt`: required string
- `session`: optional `fresh`, `resume`, or `previous`, defaults to `fresh`
- `gate`: optional object

Each `gate` supports:

- `command`: required string
- `reject_to`: required stage id that must point to an earlier stage
- `max_cycles`: required non-negative integer
- `timeout_ms`: optional positive integer, defaults to `10000`
- `env`: optional string-to-string map

Rules:

- `session: fresh` always starts a new provider session for that stage run, then overwrites that stage's current lineage so a downstream `session: previous` stage can continue the newly created session.
- `session: resume` starts fresh on the first run, then resumes that stage's prior provider session when the same stage runs again within the same execution.
- `session: previous` continues the immediately previous stage's current provider-session lineage within the same execution.
- Later `session: resume` runs fail closed if the stored provider-session handle is missing or invalid.
- `session: previous` is invalid on the first stage and also fails closed if the previous stage lineage has no stored provider-session handle.
- All stages, continuation turns, and retry attempts within one execution share the same execution workspace.
- Duplicate `stages[].id` values are invalid.
- `gate.reject_to` must refer to an earlier stage id.
- Gate exit code `0` accepts, `1` rejects and reroutes, `>=2` fails the attempt.
- Automatic retries and `start_new_attempt` reruns create a new attempt, but they restart from the failed or current checkpoint stage instead of replaying stage 1.

`session: previous` is dynamic at runtime. It always follows the immediately previous stage's resolved lineage owner, so `fresh -> previous` hands off the newly created session while chains like `resume -> previous -> previous` all continue the same underlying provider session.

### `actions`

`actions` is optional.

Each action supports:

- `id`: required string, unique within the workflow
- `label`: required string
- `command`: required string
- `confirm`: optional boolean, defaults to `false`
- `inputs`: optional array of `{ ref }` bindings
- `timeout_ms`: optional positive integer, defaults to `30000`
- `set_state`: optional string
- `start_new_attempt`: optional boolean, defaults to `false`

Rules:

- Actions are only available when the latest attempt is `completed`, the execution is not terminal, and the shared execution workspace is still retained.
- Duplicate `actions[].id` values are invalid.
- `actions[].inputs[].ref` must match a top-level input id.
- `set_state` cannot target a terminal state when `start_new_attempt` is `true`.
- If both `set_state` and `start_new_attempt` are set, the state change happens before the new attempt is created.
- When `start_new_attempt` is `true`, the new attempt reuses the same execution workspace and resumes from the failed or current checkpoint stage rather than stage 1.

### `harness`

`harness` is required when `stages` contains one or more entries. It is optional for `stages: []`. It is a generic harness declaration, not a closed union in core contracts.

Shared fields:

- `id`: required string, such as `openai/codex`, `anthropic/claude-code`, or a third-party id like `acme/review-bot`
- `auth`: required object
- `model`: optional string
- `effort`: optional string
- `turn_timeout_ms`: optional positive integer, defaults to `3600000`
- `stall_timeout_ms`: optional positive integer, defaults to `300000`
- `config`: optional object, defaults to `{}`
- `retry`: optional object with `max_retries`, `base_delay_ms`, `max_backoff_ms`
- `cancel`: optional object with `graceful_timeout_ms`, `hard_kill_timeout_ms`
- harness-specific auth/config validation comes from the installed harness manifest, not from `@workbench-ai/contracts`

If a zero-stage workflow still includes a `harness` block, Workbench keeps the shared shape and may still read generic retry or cancel defaults from it, but harness availability plus harness-specific auth/config validation do not block the workflow because no harness turn runs.

First-party harnesses currently bundled with the local platform:

#### `harness.id: openai/codex`

Codex commonly uses a secret reference auth object:

```yaml
harness:
  id: openai/codex
  auth:
    strategy: secret_ref
    ref: OPENAI_API_KEY
  model: gpt-5.4
  effort: medium
  config: {}
```

Auth strategies:

- `secret_ref`
  - resolve `auth.ref` as `OPENAI_API_KEY`
- `profile_path`
  - copy `<path>/.codex/auth.json` into the staged managed `CODEX_HOME`

Codex-specific authored config keys in `harness.config`:

- none in the first-party managed workflow surface

When `auth.strategy: secret_ref` is used, Workbench resolves `auth.ref` using this precedence:

1. the active workspace repository `.env`
2. `${WB_HOME}/.env`
3. the raw shell environment

Workflow-owned Codex turns always run in a Workbench-managed staged environment:

- `cwd = <stageSessionPath>/workspace`
- `HOME = <stageSessionPath>/home`
- `CODEX_HOME = <stageSessionPath>/home/.codex`

Workbench mirrors the live execution workspace into that staged workspace before the turn and mirrors changes back after the turn. The staged workspace excludes repo-local `.agents/` and `.codex/` so user or repo Codex state does not leak into workflow turns.

When `model` is set, Workbench passes that value through to Codex for both new and resumed headless turns. Native interactive chat launch also prefers the workflow-configured `model`; when no workflow model is configured, Codex may still fall back to the external Codex home config or cached model metadata for chat-owned launches.

When `effort` is set, Workbench passes that value through to Codex headless turns and to native interactive chat launch. The portable first-party values shared by both Codex and Claude are `low`, `medium`, and `high`. Codex additionally accepts `none`, `minimal`, and `xhigh`.

#### `harness.id: anthropic/claude-code`

Claude supports either OAuth or a secret reference:

```yaml
harness:
  id: anthropic/claude-code
  auth:
    strategy: profile_path
    path: /Users/alice/.config/claude-workflow-profile
  model: claude-sonnet-4-6
  effort: medium
  config:
    max_turns: 24
    allowed_tools:
      - Bash
      - Read
      - Write
```

Auth strategies:

- `secret_ref`
  - resolve `auth.ref` as `ANTHROPIC_API_KEY`
- `profile_path`
  - copy `<path>/.claude.json` and `<path>/.claude/.credentials.json` into the staged managed `HOME`

Claude-specific authored config keys in `harness.config`:

- `max_turns`, default `24`
- `max_budget_usd`
- `allowed_tools`
- `add_dirs`

Runtime behavior:

- Workflow-owned Claude turns always run in a Workbench-managed staged environment with `cwd = <stageSessionPath>/workspace` and `HOME = <stageSessionPath>/home`.
- Workbench mirrors the live execution workspace into that staged workspace before the turn and mirrors changes back after the turn. The staged workspace excludes repo-local `.claude/` so user or repo Claude state does not leak into workflow turns.
- With `auth.strategy: secret_ref`, Workbench resolves `auth.ref`, injects the result into `ANTHROPIC_API_KEY`, strips known provider-selector env vars, and seeds the staged managed home with the operator's ambient `.claude.json` so native Claude sessions skip first-run onboarding in isolated homes.
- With `auth.strategy: profile_path`, Workbench copies only the portable Claude auth files from the declared path. It does not copy settings, hooks, or skills.
- Workflow-owned Claude turns internally use `--permission-mode dontAsk`, `--setting-sources user`, hooks disabled, and slash commands enabled. Here `user` means the staged managed home, not the operator's real home directory.

For first-party portability, `low`, `medium`, and `high` are the shared `effort` values supported by both Claude and Codex.

Third-party harnesses follow the same structure. Core validation only requires that `harness.id` is a non-empty string and that `auth` plus `config` are objects. Availability is checked separately at runtime, so a syntactically valid workflow may still load as `unavailable` if its harness is not installed in the current runtime composition.

## Prompt Rendering

Stage prompts are rendered with Handlebars in strict mode immediately before each stage starts.

Available variables:

- `workflow`: `id`, `name`, `metadata`
- `execution`: `id`, `state`, `created_at`, `updated_at`, `triggers`
- `attempt`: `number`, `source_type`, `state`, `created_at`, `updated_at`
- `stage`: `id`, `index`, `state`
- `trigger`: the latest normalized `TriggerEvent` for the current execution turn

Important prompt rules:

- Missing variables fail the stage before the harness starts.
- There are no custom helpers registered by Workbench.
- Use dotted property access such as `{{ trigger.payload.repository }}`, `{{ execution.triggers.0.payload.message }}`, or `{{ attempt.source_type }}`.
- Interpolating a whole object such as `{{ trigger.payload }}` is not a useful pattern; Handlebars will coerce objects to a plain string representation instead of pretty-printing JSON.

## Webhook Continuations

Webhook-triggered workflows can either create or continue an execution.

- Without `execution_id`, a verified webhook creates a new execution.
- With top-level `execution_id`, Workbench appends a new trigger to that existing non-terminal execution, resumes the current attempt in place at its continuation checkpoint, and does not create a new attempt.
- During prompt rendering, `trigger` is the latest inbound event for the current turn and `execution.triggers` is the full trigger history ordered oldest to newest.

Examples:

```json
{ "message": "Start the conversation." }
```

```json
{ "execution_id": "exec_123", "message": "Continue with this follow-up." }
```

## Command Environment Contracts

### Input resolvers

Command-backed `select` inputs are resolved before the user submits a manual run or action.

Manual input resolvers run from `WB_PROJECT_ROOT`. Action input resolvers run from the execution workspace so they can inspect the live checkout. Both scopes receive:

- `WB_PROJECT_ROOT`
- `WB_RUNTIME_URL`
- `WB_OWNER_ID`
- `WB_OWNER_KIND` as `workflow`
- `WB_INPUT_SCOPE` as `manual` or `action`
- `WB_INPUT_ID`
- `WB_INPUT_VALUES_JSON`
- `WB_INPUT_VALUE_<ID>` for each already known input value

Action-scope input resolvers also inherit the same execution context the eventual action command receives, including:

- `WB_ACTION_ID`
- `WB_EXECUTION_ID`
- `WB_LATEST_ATTEMPT_NUMBER`
- `WB_WORKSPACE_PATH`
- `WB_TRIGGER_FILE`
- `WB_TRIGGER_TYPE`

### Hooks

Hooks always receive:

- `WB_PROJECT_ROOT`
- `WB_RUNTIME_URL`
- `WB_OWNER_ID`
- `WB_OWNER_KIND`
- `WB_EXECUTION_ID`
- `WB_ATTEMPT_NUMBER`
- `WB_WORKSPACE_PATH`
- `WB_TRIGGER_TYPE`
- `WB_TRIGGER_FILE`

If `hooks.env` is set, those entries are merged into the hook process environment.

`WB_RUNTIME_URL` is the canonical base URL for the running Workbench runtime. Workflow-authored self-chaining should use the first-party manual routes through that base URL, for example `POST "$WB_RUNTIME_URL/api/v1/workflows/<target>/executions"` to enqueue a different workflow or `POST "$WB_RUNTIME_URL/api/v1/executions/$WB_EXECUTION_ID/triggers/manual"` to append another turn to the current execution. Prefer those routes over the signed webhook route for internal chaining.

### Gates

Gates receive the hook variables plus:

- `WB_STAGE_ID`
- `WB_STAGE_RUN_INDEX`
- `WB_STAGE_OUTPUT_FILE`
- `WB_STAGE_EVENTS_FILE`

The final assistant output for the stage is provided to the gate command on `stdin`.

### Actions

Actions receive:

- `WB_ACTION_ID`
- `WB_ACTION_INPUTS_JSON`
- `WB_ACTION_INPUT_<ID>` for each submitted action input value
- `WB_PROJECT_ROOT`
- `WB_RUNTIME_URL`
- `WB_OWNER_ID`
- `WB_OWNER_KIND`
- `WB_EXECUTION_ID`
- `WB_LATEST_ATTEMPT_NUMBER`
- `WB_WORKSPACE_PATH`
- `WB_TRIGGER_FILE`
- `WB_TRIGGER_TYPE`

Resolved action inputs are provided on `stdin` as JSON:

```json
{
  "inputs": {
    "target_branch": "main"
  }
}
```

## Runtime Semantics Authors Should Know

- Existing valid workflows stay active when a later on-disk edit becomes invalid. The record becomes `validity_status: "invalid"` and keeps the last valid `effective_workflow`.
- Valid workflows may still load as `availability_status: "unavailable"` when the current runtime composition lacks the declared harness, workspace mode, or required capability.
- The raw YAML draft remains authoritative in the web editor, so inline errors operate on `source_yaml`, not the last valid effective parse.
- Save-time validation is collection-aware. A file can be individually valid but still rejected if another file already uses the same effective workflow id.
- If one file edits itself into another file's effective workflow id, the unchanged owner stays loaded and the edited file becomes invalid under its prior file-backed record.
- `metadata.enabled` defaults to `true`. A workflow with no `metadata` block is still runnable when otherwise valid.
- Manual runs and cron runs always create a new execution. Webhook continuation semantics are documented in [Webhook Continuations](#webhook-continuations).
- Automatic retries and action-triggered reruns create a new attempt on the same execution.
- Attempt runtime state and execution business state are different concepts. Attempt state tracks runtime lifecycle; `state` in the workflow tracks board and review semantics.

## Web Editor Behavior

The `/workflows` modal is a Monaco-backed YAML editor. It edits `source_yaml` directly, validates the current draft before save, and preserves any supported or unsupported sections because it does not translate through a second authoring format.

The Monaco editor uses the same shared structural validator as the server save path and `workbench validate --schema-only`, so inline markers and save blocking come from the same rules.

## Validator Usage

Use the shared validator before committing or saving generated YAML:

```bash
workbench validate
workbench validate --schema-only
workbench validate .workbench/workflows
workbench validate .workbench/workflows/review.yaml
workbench validate docs/examples/workflows --schema-only
workbench validate --json
```

The legacy `pnpm workflow:validate ...` wrapper still works inside this source checkout, but `workbench validate` is the canonical interface for repo-local and packaged use.

Default `workbench validate` behavior:

- uses the shared contracts validator for structural YAML parsing and duplicate-id checks
- then checks runtime readiness for every structurally valid workflow in the current local platform
- fails when a workflow is valid but unavailable on that local platform

Use `--schema-only` when you want only the structural answer.

Target behavior:

- file target: validates that one YAML file
- directory target with `.workbench/workflows`: validates `.workbench/workflows/*.yaml`
- directory target without `.workbench/workflows`: validates direct `*.yaml` files in that directory as one collection
- directory target inside a git repo with no direct YAML files: validates that repo's `.workbench/workflows`

Exit codes:

- `0`: all workflows valid, and in default mode also runtime-ready
- `1`: validation failure or runtime error
- `2`: CLI usage error

The web editor and server save path use the same structural validator as `workbench validate --schema-only`, so a file that passes `--schema-only` should produce the same answer in Monaco and on create or update. Plain `workbench validate` is stricter because it also checks local runtime readiness.

## Troubleshooting

- `Unknown key`: move repo-local data under `metadata` or rename the key to `x-*`.
- `workspace.root has been removed`: delete `workspace.root`; Workbench now manages execution-workspace location outside the project root.
- `duplicate_workflow_id`: change `metadata.id` or rename the file so the effective ids are unique.
- harness unavailable: install the declared `harness.id` in the current runtime or start Workbench with a harness list that provides it.
- missing harness readiness: satisfy the harness's local auth or readiness requirement, or rerun with `--schema-only` when you only need structural validation.
- `harness.auth.*` / `harness.config.*` error: the installed harness manifest rejected the provided auth or config object.
- web editor save blocked: the current YAML is invalid. Use the validator output in the modal and rerun `workbench validate --schema-only` if you want the same structural answer at the CLI.

## LLM Authoring Checklist

When generating workflow YAML, always:

- emit exactly one YAML document
- include every required top-level section
- choose `harness.id` explicitly
- include `harness.auth` whenever the workflow has stages, and set `harness.config` only for provider-specific fields that still matter
- use logical secret refs instead of embedding secret values
- use `triggers: {}` for manual-only workflows
- keep stage ids and action ids unique
- validate with `workbench validate` before returning the final YAML
